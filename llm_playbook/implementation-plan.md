====================================================

STATUS: DRAFT r3. Lineage: r1 (initial plan) -> r2 (self-review, findings
RF1-RF8) -> fresh-instance adversarial review (findings AF1-AF13) ->
workflow interview (10 questions answered) -> this document. r3 is
self-contained: no section requires r1 or r2 to interpret (this was
itself review finding AF13).

DECIDED: the toolkit lives as a self-contained directory inside the
the parent repo repo (the author's existing monorepo of projects), not as a
standalone repo. Promotion to its own repo is a reserved later move.
Directory name is still undecided (open fork F2); TOOLKIT is the
placeholder throughout. All toolkit files are ASCII only.

WORKFLOW GROUND TRUTH (from the interview; several design choices below
depend on these facts, so they are recorded here and any change to them
is a tripwire, see ADR list in T-002):
  W1. All work happens in web and desktop chat interfaces. No agentic or
      CLI environments (tools that automatically read a CLAUDE.md or
      AGENTS.md file at a project root) are in use yet.
  W2. Context is pasted or pointed-to ONCE at thread kickoff and never
      refreshed mid-thread. In Claude chats the model has repo access
      and can be pointed at files; in other chats the thread must finish
      within one context window.
  W3. Threads run ONE AT A TIME. Parallel execution (git worktree plus
      tmux or similar) is an ambition, not a practice.
  W4. Threads are mostly role-pure (a thread is one conversation with
      the model, playing one role: design, implementation, or capture).
      The known exception: a thread scoped as design absorbs the
      implementation when the fix turns out to be trivial. The author's
      stated tendency is to EXTEND thread scope when the model flags
      adjacent work; this is treated as a tendency to counteract.
  W5. Threads that die are almost always idea/design threads that never
      reached implementation; the author usually senses this coming.
      Mid-implementation abandonment is rare.
  W6. Consumer projects live in three transport situations: inside
      the parent repo, in their own GitHub repos, and in private local git
      repos with no online remote (content must be extracted or
      archived to reach a chat).
  W7. The author prefers friction automated away NOW via simple scripts
      and hooks, rather than documented manual steps.
  W8. kbd ("knowledge base desk", a playful name) is a real directory
      at $HOME/personal_repos/usb-repos/kbd, OUTSIDE the parent repo. It
      is the destination for study/reference material split out of
      project repos.

r3 CHANGES FROM r2 (each traces to a review finding AF# or interview
answer W#):
  * Render priority inverted (AF-corrected prediction, W1/W2): the
    primary render target is a pasteable CONTEXT.md at each project
    root; CLAUDE.md/AGENTS.md become secondary GENERATED outputs that
    future-proof an eventual agentic transition. "Render" throughout
    means: a document generated by hand from toolkit preference sources,
    carrying a version-and-hash stamp, that delivers those preferences
    into a chat.
  * Entry files collapsed from three role files to ONE file with a
    common core plus role sections (AF2, W4). A scope-extension
    counter-instinct rule is added to the kickoff skeleton (W4).
  * Terminal thread states refined (AF8, W5): "parked" added for design
    threads that never reached implementation; their existing design
    artifacts count as the close artifact. "abandoned" reserved for
    mid-implementation death and keeps its mandatory close artifact.
  * A scripts/ directory is born at T-001 (AF10, AF11, W7): one check
    script (line budgets, token estimates, containment grep) installed
    as a pre-commit hook, plus bootstrap.sh added before G-1 (AF6, W7)
    which performs the sparse checkout at a recorded commit SHA and
    emits the kickoff paste block.
  * SHA pinning is now real, not forensic (AF6): bootstrap.sh checks
    out the recorded SHA. One-thread-at-a-time (W3) makes the
    hot-fix-to-open-threads caveat moot; a tripwire ADR records which
    assumptions break if concurrency exceeds one.
  * Directory taxonomy declared cosmetic (AF1): protocol/ and
    preferences/ are shelving only; MANIFEST.md is the sole
    authoritative grouping of documents into load classes; documents
    may never reference each other by directory semantics.
  * Precedence doc restructured (AF3) into render-time vs runtime
    precedence; render stamp extended with a content hash (AF4) so
    hand-edited renders are detected mechanically at kickoff.
  * Overlay composition set at ITEM level, not layer level (AF7, W6):
    preference items get stable ids; a private item id shadows the
    public item with the same id.
  * Private-repo transport recognized as its own small deliverable
    (W6), separate from the preferences overlay: transport.md plus an
    archive/extract helper.
  * Gate restructure (AF5): drill (the first consumer project) adopts
    the skeleton immediately after the skeleton lands, a live-thread
    gate G-1.5 runs BEFORE prompts are written, and G-2 narrows to
    testing bootstrap plus prompts only. Every gate now states its
    failure branch (which commits reopen).
  * bundle/SPEC.md demoted to phase V (AF9): the render plus the entry
    file already serve the paste-into-chat consumer; a bundle returns
    only as generated output of the v2 builder if that pair proves
    insufficient.
  * decisions/index.md dropped until a second era shard exists (AF12).
  * Disposition map fully inlined; kbd expanded (AF13, W8).

COMPLEXITY CLASSES (rubric candidate, formalized in T-008):
  [S] small  -- one file, mostly transcription of already-agreed content
  [M] medium -- one file, requires generalization or new synthesis
  [L] large  -- multi-file or requires splitting an existing doc

STATELESSNESS BUDGETS (acceptance criteria; "statelessness" here means
every thread starts with a model that remembers nothing, so all context
must arrive through documents whose size is priced in tokens; enforced
mechanically by the check script from T-001 on):
  * Any single protocol/ or preferences/ doc: <= 200 lines.
  * The entry file: core plus largest role section <= 150 lines,
    pasteable whole into a plain chat.
  * A rendered CONTEXT.md: <= 250 lines.
  * Total required-read set per role (the ordered document list a
    thread of that role must ingest): DESIGN <= ~10k tokens,
    IMPLEMENTATION <= ~6k, CAPTURE <= ~6k. Token estimate: characters
    divided by 4; budgets carry headroom so roughness is acceptable.

completes from project state before pasting). Gates (G-*) are
mandatory pauses with stated failure branches, not commits.

T-004 [LANDED] [S] protocol/precedence.md (restructured per AF3)
  TWO chains, explicitly separated. Render-time and authoring-time
  precedence: live human > project instance > toolkit > model
  defaults -- resolved by the HUMAN when a render is built or a doc

  full stop -- the model never re-resolves the longer chain because
  the render already collapsed it. Worked example walks both: an
  instance rule contradicts a toolkit rule -> the render emits the
  instance rule; mid-thread the human contradicts the render -> the
