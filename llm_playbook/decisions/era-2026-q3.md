ADR-001 directory name: llm_playbook
Context: fork F2. Constraints: short; not "llm" (collides with the
per-project instance directory convention, ADR-017); the monorepo
root already contains an unrelated "llms" directory to avoid.
Decision: llm_playbook.
Alternatives: playbook (rejected: less self-describing at the
monorepo root); toolkit (rejected: generic); llm (rejected by
constraint).

ADR-002 placement inside the parent monorepo; promotion reserved
Context: the playbook could be its own repository from birth.
Decision: it lives as a self-contained directory inside the author's
existing monorepo. Promotion to a standalone repository is a
reserved later move: because the tree is self-contained, promotion
is a copy or a git filter-repo run, and the raw URLs in MANIFEST.md
are the only content change. The containment checks in
scripts/check.sh (no upward path references, no parent-repository
name in prose outside MANIFEST raw URLs) exist to keep this move
cheap; one upward reference would otherwise fail silently until
promotion day.
Alternatives: standalone repo now (rejected: overhead before the
toolkit has proven itself on a second consumer).

ADR-003 directory names are cosmetic; MANIFEST is authoritative
Context: review finding AF1 -- the directory taxonomy cut on the
wrong axis; grouping by directory would acquire false semantics.
Decision: protocol/ and preferences/ are shelving only. MANIFEST.md
is the sole authoritative grouping of documents into load classes
(load-always, load-per-role, on-demand) and the sole carrier of
per-role required-read lists. Documents never reference each other
by directory semantics; cross-references go by filename.
Alternatives: semantic directories (rejected per AF1).

ADR-004 sparse-clone bootstrap with real SHA pinning; read-only
Context: review finding AF6 -- SHA recording in r2 was provenance
theater: the SHA was written down but nothing checked it out.
Decision: scripts/bootstrap.sh (T-007b) performs the sparse checkout
of llm_playbook plus one project directory AT the recorded SHA, so
the SHA is ground truth rather than forensic decoration. During
v0.x each thread records and checks out a playbook SHA; pinning by
version tag begins at v1.0. The playbook checkout is read-only from
a thread's point of view: refinements land project-side (in the
project's refinements file) and reach the playbook only through a
human editorial pass.
Alternatives: track main and merely record the SHA (rejected per
AF6); tag-pinning from birth (rejected: tags do not exist yet).

ADR-005 project-prefixed thread ids: PROJ-ROLE-NNN
Context: fork F1; ids must be unique across projects and legible in
a close artifact with no surrounding context.
Decision: PROJ-ROLE-NNN, e.g. DRILL-IMPL-004. Grammar formalized in
naming.md (T-003).
Alternatives: PROJ-NNN (rejected: role is load-bearing for the
role-purity data collected at V-207).

ADR-006 one entry file with role sections
Context: review finding AF2 -- three role entry files were designed
on evidence from a single consumer; interview answer W4 -- threads
are mostly role-pure but a design thread may absorb a trivial fix.
Decision: ONE file, entry/ENTRY.md: a common core carrying the
invariants every thread needs, plus three delimited role sections
(DESIGN, IMPLEMENTATION, CAPTURE). Delimitation makes a future
split mechanical if role-purity data (V-207) warrants it; merging
three files later would orphan kickoff skeletons. Splitting is
cheap, merging is not.
Alternatives: three files (rejected per AF2).

ADR-007 prompts are standalone documents
Context: prompts could reference protocol documents for shared text.
Decision: each of the five ported prompts is standalone: pasteable
alone into a chat, with FILL-IN slots named after skeleton
STATUS.md headings. Duplication between prompts is accepted as the
price of statelessness.
Alternatives: prompts including shared fragments (rejected: a
stateless chat cannot resolve includes).

ADR-008 item-level overlay shadowing; item ids LAYER-NNN
Context: review finding AF7 -- layer-granularity overlays force a
private user to fork a whole layer to change one item; fork F6.
Decision: preference layers are composed of self-contained ITEMS
with stable ids of the form LAYER-NNN (e.g. CONSTRAINT-004). A
private overlay item with the same id replaces the public item;
layers are never forked whole. Ids are stable forever and never
reused after retirement.
Alternatives: layer-level overlays (rejected per AF7).

ADR-009 public layers plus a private-overlay slot
Context: some preferences are private to a machine or an employer
and cannot live in a public repository.
Decision: the public layers file defines all items; a designated
private overlay location may shadow items by id (ADR-008). The
render pass composes public plus overlay. Transport of private
CONTENT is a separate concern (transport.md, T-015b) and neither
mechanism references the other.
Alternatives: a single private fork of the layers (rejected:
diverges silently from the public source).

ADR-010 classify-first kbd ingestion
Context: interview answer W8 -- kbd ("knowledge base desk") is a
real directory at $HOME/personal_repos/usb-repos/kbd, outside the
monorepo, the destination for study and reference material split
out of project repos.
Decision: material bound for kbd is first marked in place with a
destination header (phase D-104), then moved in a later two-pass,
classify-first ingestion (V-203). Nothing moves unclassified.
Alternatives: move immediately (rejected: ingestion criteria do not
exist yet).

ADR-011 lite mode
Context: a full pipeline on a one-paragraph task is ceremony.
Decision: a task describable completely in one paragraph with no
design fork may skip the pipeline: state the task, cite the style
contract, land it. The threshold lives in README.md; anything
larger enters through the entry file.
Alternatives: pipeline always (rejected: friction the author will
route around, which is worse than sanctioned lite mode).

ADR-012 accretion countermeasures; index deferred to shard two
Context: review findings RF3 and AF12 -- meta-work is an attractor,
and an index file was bookkeeping reintroduced by an
anti-bookkeeping measure.
Decision: decisions are sharded by era (this file is the first
shard); archive/ exists from birth and superseded documents move
there, never deleted. index.md is created together with the second
era shard, not before; until then the single shard indexes itself.
Alternatives: index from birth (rejected per AF12); unsharded
decisions file (rejected: unbounded growth).

ADR-013 render priority: CONTEXT.md first; corrected prediction
Context: r2 asserted CLAUDE.md auto-ingestion as the primary
zero-token render channel. The workflow interview (W1) showed no
agentic environment is in use: all work happens in web and desktop
chat, and context arrives once at kickoff (W2). The r2 assertion
is preserved here as a worked example of legislating for an
imagined workflow.
Decision: the primary render target is CONTEXT.md at each project
root -- pointed at in repo-access chats, pasted whole elsewhere.
CLAUDE.md / AGENTS.md and the platform settings block are
SECONDARY, generated from the same sources in the same render pass
at zero marginal maintenance, future-proofing an agentic
transition. Every render carries the stamp "rendered from playbook
vX.Y <sha> content-hash <8 chars>" plus a DO-NOT-EDIT header;
renders are regenerated, never patched (AF4).
Alternatives: CLAUDE.md primary (rejected by W1: the consumer does
not exist).

ADR-014 spike outcomes recorded as SPECULATIVE
Context: several design choices rest on predictions, not lived use.
Decision: any predicted outcome is marked SPECULATIVE where
recorded. Deferred spikes (V-205) run only when a recorded
SPECULATIVE prediction is contradicted by lived experience.
Alternatives: silent predictions (rejected: uncheckable later).

ADR-015 wrong-bottleneck prediction and redirect condition
Context: review finding RF7.
Decision: recorded prediction (SPECULATIVE): stale instance ground
truth -- project STATUS and instance docs lagging reality -- will
dominate failures, not preference drift. Redirect condition: if
lived threads confirm this, escalate V-204 (render generation and
FILL-IN substitution automation) ahead of other deferred work.
Alternatives: none; this entry exists to make the prediction
falsifiable.

ADR-016 concurrency tripwire
Context: interview answer W3 -- threads run one at a time; parallel
execution is an ambition, not a practice.
Decision: recorded tripwire: one thread at a time (W3) is
load-bearing for: (a) hot-fixes reaching open threads via the live
human channel only, (b) render hash check at kickoff only, (c) no
push-freeze convention needed; concurrency greater than one
invalidates all three together -- revisit them as a set (V-207).
Alternatives: designing for concurrency now (rejected: imagined
workflow, see ADR-013's lesson).

ADR-017 instance directory name stays llm/
Context: fork F3. Consumer projects already carry their instance
documents (STATUS, decisions, refinements, handoffs) in a directory
named llm/.
Decision: keep llm/ as the per-project instance directory name. The
playbook directory name avoids colliding with it (ADR-001).
Alternatives: rename instance dirs (rejected: churn in consumer
repos for no gain).

ADR-018 dates in frontmatter, not filenames
Context: fork F5. Dated filenames rot and break references when
content is revised.
Decision: dates live in document frontmatter only; filenames are
dateless. Exception: decisions era shards, whose era is the
filename (this file).
Alternatives: dated filenames (rejected: reference rot).

ADR-019 LICENSE deferred
Context: the plan's T-001 contents include a LICENSE file; the
author ruled in-thread (PLAYBOOK-IMPL-001) that no license is added
until its legal implications are considered.
Decision: no LICENSE file in v0.x. The plan item stays open;
revisit at promotion (ADR-002) at the latest, since a standalone
public repository without a license has a different default legal
posture than a directory in a personal monorepo.
Alternatives: MIT now (drafted, then withdrawn by author ruling).
