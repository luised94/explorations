ADR-001 directory name: llm_playbook
Context: fork F2. Constraints: short; not "llm" (collides with the
per-project instance directory convention, ADR-017); the monorepo
root already contains an unrelated "llms" directory to avoid.
Decision: llm_playbook.
Alternatives: playbook (rejected: less self-describing at the
monorepo root); toolkit (rejected: generic); llm (rejected by
constraint).

ADR-002 placement inside the parent monorepo; promotion reserved
Context: the playbook could be its own repository from birth.
Decision: it lives as a self-contained directory inside the author's
existing monorepo. Promotion to a standalone repository is a
reserved later move: because the tree is self-contained, promotion
is a copy or a git filter-repo run, and the raw URLs in MANIFEST.md
are the only content change. The containment checks in
scripts/check.sh (no upward path references, no parent-repository
name in prose outside MANIFEST raw URLs) exist to keep this move
cheap; one upward reference would otherwise fail silently until
promotion day.
Alternatives: standalone repo now (rejected: overhead before the
toolkit has proven itself on a second consumer).

ADR-003 directory names are cosmetic; MANIFEST is authoritative
Context: review finding AF1 -- the directory taxonomy cut on the
wrong axis; grouping by directory would acquire false semantics.
Decision: protocol/ and preferences/ are shelving only. MANIFEST.md
is the sole authoritative grouping of documents into load classes
(load-always, load-per-role, on-demand) and the sole carrier of
per-role required-read lists. Documents never reference each other
by directory semantics; cross-references go by filename.
Alternatives: semantic directories (rejected per AF1).

ADR-004 sparse-clone bootstrap with real SHA pinning; read-only
Context: review finding AF6 -- SHA recording in r2 was provenance
theater: the SHA was written down but nothing checked it out.
Decision: scripts/bootstrap.sh (T-007b) performs the sparse checkout
of llm_playbook plus one project directory AT the recorded SHA, so
the SHA is ground truth rather than forensic decoration. During
v0.x each thread records and checks out a playbook SHA; pinning by
version tag begins at v1.0. The playbook checkout is read-only from
a thread's point of view: refinements land project-side (in the
project's refinements file) and reach the playbook only through a
human editorial pass.
Alternatives: track main and merely record the SHA (rejected per
AF6); tag-pinning from birth (rejected: tags do not exist yet).
SUPERSEDED IN PART by ADR-021 (checkout mechanism retired); the
SHA-as-ground-truth principle stands, now satisfied by the packer
recording and echoing the SHA.

ADR-006 one entry file with role sections
Context: review finding AF2 -- three role entry files were designed

are mostly role-pure but a design thread may absorb a trivial fix.
Decision: ONE file, entry/ENTRY.md: a common core carrying the

ADR-007 prompts are standalone documents

Alternatives: layer-level overlays (rejected per AF7).

mechanism references the other.
Alternatives: a single private fork of the layers (rejected:

ADR-020 CONTEXT.md file located in the project's llm/ instance dir
Context: G-1 finding. The plan (T-007, render.md, ADR-013) placed the
CONTEXT.md file at "project root". Running against drill in the
monorepo showed root placement is wrong: multiple projects share the
monorepo tree, so a root-level CONTEXT.md is ambiguous and
collision-prone, and the per-project instance directory llm/ (ADR-017,
fork F3) is already the home for instance state (STATUS.md,
decisions.md, refinements.md). CONTEXT.md is instance state.
Decision: the committed CONTEXT.md file is located at
<project>/llm/CONTEXT.md, beside the other instance docs. It is
AUTHORED/UPDATED BY HAND from the preference sources -- this manual
authoring is what the plan loosely calls "rendering"; there is no
automated render generator in v0.1 (that is V-204, deferred). The only
mechanical step today is the version+content-hash stamp the author
runs and pastes into the header. The file is committed instance state
and serves the NEXT thread. For a standalone external repo (W6 case 2)
a README pointer supplies discoverability.
Alternatives: project root (rejected: monorepo collision; the original
wording imagined standalone repos); a dedicated render/ subdir
(rejected: llm/ already means "instance state").

ADR-021 transport is archive-or-paste, scratch-only, committed-read
Context: G-1 finding. ADR-004 specified bootstrap.sh doing a local
sparse-checkout into a new directory. The real transport need (W2, W6)
is getting content INTO a chat, not materializing another on-disk
copy; one thread at a time (W3) makes the second copy worthless.
Decision: transport is archive-or-paste.
  - Archive mode: git archive --format=tar.gz of the chosen (possibly
    SUBSET) file set at HEAD; the SHA is recorded by the author at pack
    time and echoed in the output.
  - Paste mode: the same file set composed into one text block written
    to a temp file; with an --edit/-e flag the temp file opens in nvim
    before the author copies it out; without the flag the path is
    printed.
  - SCRATCH-ONLY: the pack step writes ONLY to scratch space (/tmp). It
    NEVER writes into the git tree. It produces nothing the repo keeps.
  - COMMITTED-READ: the pack step reads only COMMITTED files (git
    archive sees HEAD). Consequence: CONTEXT.md must be committed
    BEFORE packing, or the artifact will not contain it. Documented in
    transport.md. Committed-only reading keeps the recorded SHA honest.
  - Composition is cheap, idempotent, reproducible: byte-identical
    output for the same SHA and inputs; never appends or mutates in
    place; always regenerates from source.
  - Selective packing (a subset of files a thread needs) is a
    first-class mode.
Alternatives: keep local sparse-checkout (rejected: wrong mechanic,
W3); ship .git / git bundle for real patches (rejected: huge monorepo
.git, leaks unrelated history, fights lightweight transport -- see
ADR-023); let paste mode read the working tree (rejected:
committed-only keeps the SHA honest and both paths uniform).
SUPERSEDES the checkout MECHANISM in ADR-004. The SHA-as-ground-truth
principle of ADR-004/AF6 stands, now satisfied by the packer recording
and echoing the SHA.

ADR-022 base-plus-overlay composition happens at transport time
Context: the plan had item-level overlay shadowing as a preferences
concept (ADR-008, AF7) but never said WHERE composition occurs. The
author's model: a pack is "playbook base plus project overlay",
composed, not just a pile of files. The playbook (llm_playbook/) files
are the default base layer; the project-specific instance files layer
on top; private overlay items shadow public items by id (ADR-008).
Decision: composition is performed at transport (pack) time. The pack
step assembles the base, applies the project overlay, and resolves
item-level shadowing, emitting the composed result as archive or
paste. The thing pasted or uploaded is already the composed
base-plus-overlay, so the stateless reader receives resolved
preferences, not raw layers to merge mentally.
Staging (W4 scope-counteraction): implement archive-or-paste FIRST
(ADR-021), add base-plus-overlay composition as a distinct SECOND
increment. Do not bundle the two into one change.
Alternatives: compose in the render step only (rejected: transport
also carries protocol/ base docs, so composition belongs at the
transport boundary that sees both); leave composition to the human or
model each time (rejected: not reproducible, defeats idempotency).

ADR-023 MODE A delivery is apply-and-commit-locally, not git-am
Context: G-1 / transport finding. The kickoff's MODE A said "deliver
each commit as a git-am-able patch." This is incompatible with archive
transport (ADR-021): git archive strips .git, so the model's sandbox
holds files with no history, no HEAD, no ancestry. A patch generated
from a throwaway git init has a fabricated unrelated root; git am on
the author's side would reject it or apply against the wrong base and
corrupt provenance -- the "provenance theater" AF6 warns against.
Decision: MODE A delivery is file-content-plus-apply-and-commit-locally.
  - New files (most of the T-008..T-015b range): the model supplies the
    complete file content, the exact target path, and a suggested
    complete commit message. The author saves, git add, git commit.
  - Edits: the model supplies a unified diff (git apply-able) plus a
    suggested commit message. The author runs git apply, reviews,
    commits.
  - Either way the model supplies a complete suggested commit message
    (more consistent and convention-conformant than by hand; messages
    follow the commit-prefix grammar in naming.md).
  - The AUTHOR is the committer, in the real repo, against real HEAD.
    Correct: the playbook checkout is read-only on the model side
    (ADR-004) and changes land project-side.
Alternatives: ship .git so real patches work (rejected, see ADR-021);
editor-instructions instead of diffs for edits (rejected: diffs are
testable via git apply and unambiguous).
SUPERSEDES the "git-am-able patch" wording in the kickoff/entry
WORKING METHOD line.

ADR-025 toolkit base context lives as layer items, not a new file
Context: a default/base context is wanted -- standing text about the
user and the workflow, shared across all projects, distinct from each
project's specific CONTEXT.md. It could be a new file, but the plan
already has a Persona layer (ADR-008, T-005) and an item-id plus
shadowing model (ADR-022) that fit it exactly.
Decision: the toolkit base context lives as ITEMS in layers.md, not as
a new file. User-identity facts become Persona items; workflow ground
truth (W1-W8) becomes Constraint/Convention items; each carries a
LAYER-NNN id. These items form the BASE layer that composition
(ADR-022) places UNDER each project's CONTEXT.md overlay; a private
overlay may shadow any of them by id. The "state the task, ask if
unclear" instruction is NOT context content -- it is a kickoff
invariant and lives in ENTRY.md (T-014).
Alternatives: a new preferences/context-base.md (rejected: its
relationship to layers.md would be fuzzy and it would sit outside the
item-id composition model); put the ask-the-task instruction in the
context items (rejected: it is a kickoff mechanic, belongs in
ENTRY.md).
Constraint: layers.md must stay within the 200-line statelessness
budget (check.sh-enforced when the hook is installed); if the added
items breach it, that is a finding, not a silent overflow.

Alternatives: write the rule now (rejected: entangled with the

ADR-027 commit grammar has no plan-meta form
Context: the COMMIT PREFIXES rule in naming.md assumes work carries a
plan-id ("<proj>: <plan-id> <summary>"). Work ON the plan document
itself -- renaming it, recording findings that restructure it -- has
no plan-id, because it precedes or reshapes the plan-ids it would
reference. The Stage A commits of PLAYBOOK-DESIGN-002 are exactly this
case.
Decision: accept a plan-id-less summary form for plan-meta commits for
now ("playbook: <summary>" with no plan-id). Recorded as a grammar gap;
revisit with a dedicated meta-prefix if plan-meta work recurs often
enough to warrant one.
Alternatives: invent a meta-prefix now (rejected: one instance is thin
evidence, W4 counsels against generalizing early); force a nearest
plan-id (rejected: misattributes the work).

ADR-028 check.sh budgets provisional; rework deferred
Context: the line ceilings (200/150/250) and token budgets in check.sh
were hand-set without real line-count data, and the containment grep
is aggressive: a case-insensitive match on the parent-repo name will
flag any prose mention, including unavoidable ones. The pre-commit
hook is also NOT currently installed, so none of check.sh's rules bite
at present.
Decision: treat the budgets as PROVISIONAL placeholders and defer the
check.sh rework. When reworked: recalibrate ceilings against measured
document sizes, and narrow the containment grep (e.g. exempt documents
that legitimately quote the parent name, as MANIFEST is exempted).
Install the pre-commit hook (T-001 accept criterion) as part of that
rework. Until then, checks are run manually and do not gate.
Alternatives: rework now (rejected: no measured data yet; mid-effort
churn); delete the budgets (rejected: they are a real accretion
countermeasure, just uncalibrated).

ADR-029 implementation-status convention
Context: marking every completed commit DONE in the plan with landing
detail would bloat the plan into a massive file over time -- the
accretion the playbook fights elsewhere. But deleting completed
plan-ids breaks the ADR<->plan-id references the decision record
depends on (e.g. "supersedes T-007b").
Decision: the PLAN marks landed plan-ids TERSELY (a [LANDED] token, no
paragraph); the plan-id stays so references resolve. Live state --
which plan-ids are done, the current SHA baseline, open threads, next
action -- lives in STATUS.md (created at T-011) and points back to the
plan. Per-thread detail lives in close artifacts
(close-PROJ-ROLE-NNN.md). Until T-011 lands a committed STATUS.md, the
terse plan marks are the sole record; this is a known temporary gap.
Alternatives: mark-DONE-with-detail in the plan (rejected: bloat);
delete completed items (rejected: dangling references).

ADR-030 a kickoff names the render and states that it binds
Context: ADR-020 placed the render at <project>/llm/CONTEXT.md and
stopped there. G-1b showed placement is not enough. A design thread was
given the whole drill archive -- 115 files, 1.7 MB, on the order of
400k tokens -- and never opened the render. It found and followed
drill's own CODING_CONVENTIONS.md instead, by its own initiative. Six
of seven style checks still passed, sourced from that competing
document and from the surrounding code, so the pass rate concealed a
render with zero influence. An archive delivers everything and signals
nothing; a paste block buries the render mid-stream. The defect is not
transport, which worked correctly throughout, and not render content,
which was well-formed and within budget. Nothing told the thread it was
BOUND by anything. render.md compounded this by referring to a "kickoff
skeleton" that was never defined anywhere in the playbook.
Decision: every kickoff NAMES the render and states that it binds the
thread. One sentence, both delivery modes -- in paste mode the render
is in the block, in archive mode the kickoff says to unpack and read it
first. The kickoff also states that other documents in the repository
do not govern, and that where any document disagrees with the render
the render wins and the thread flags the disagreement. The recurring
text becomes a playbook artifact, protocol/kickoff.md, holding a
KICKOFF template and a HANDOFF template rather than prose retyped per
thread; it also supplies the previously-undefined kickoff skeleton that
render.md's stamp statement depends on. Handoff filenames follow
naming.md's handoff-<FROM>-to-<TO>.md.
Alternatives: a discoverability flag on pack-repo.sh (rejected: it
couples transport to protocol, so every later protocol change would
require a packer change, and it adds per-invocation configuration that
can be set wrongly or forgotten -- the packer moves bytes); having the
packer emit a prompt file (rejected for the same coupling); relying on
placement alone (rejected: this is what failed).
Consequence: threads must not be told they are being evaluated when
this is tested, and the diagnostic question ("what in your context told
you this -- quote it") replaces the checklist, since a checklist cannot
distinguish a render that landed from priors that happened to agree.

ADR-031 render ceiling is tier-dependent, not absolute
Context: the 250-line render ceiling (T-007, enforced by check.sh on
staged CONTEXT.md) exists to bound a PASTED block, where the render
competes for a single context window. Archive delivery changes the
economics. The evaluating thread needed four separate regions of a
2431-line module and retrieved each in one targeted call; a 250-line
window would have forced repeated reconstruction or a lossy summary.
Authoring drill's render under the ceiling already meant compressing
288 lines of source into 198.
Note on the smaller-and-denser versus larger-and-complete question:
G-1b is weak evidence either way. The 198-line render was already
compressed and was never opened at all, so compression was not the
failure mode and density was never tested. That question is left open
here rather than settled by a result that does not bear on it; the
tiering below does not depend on its answer, because it changes WHERE
the ceiling binds rather than how dense the render should be.
Decision: the ceiling becomes TIER-DEPENDENT -- BINDING for paste
delivery, ADVISORY for archive delivery -- rather than absolute. In
paste mode the render competes for a single context window and a
400k-token block is impossible, so the bound is real and enforced. In
archive mode the thread has a shell and targeted search beats any
fixed-size excerpt, so the ceiling is guidance and a render that
exceeds it is not a defect. check.sh keeps enforcing the ceiling
where it binds; a render authored for archive delivery that exceeds
250 lines is recorded as a deliberate tier choice, not a violation.
Alternatives: drop the ceiling entirely (rejected: it loses the
constraint exactly where it still bites, since paste mode cannot
exceed a context window); keep it absolute (rejected: it optimizes
for a delivery mode the sandbox makes unnecessary, at a real cost in
fidelity).

ADR-032 pack is primary for reading; a checkout remains correct for
executing (clarifies ADR-021)
Context: ADR-021 retired the sparse-checkout bootstrap in favour of
archive-or-paste. Its index line reads "transport is archive-or-paste,
scratch-only, committed-read", which is broader than the decision was:
the argument was that copying a tree merely to READ it is waste when
one thread runs at a time. IMPL-003 misread its own decision in a
first-draft handoff and called drill's clone-and-verify prompt obsolete
because it sparse-clones. That was wrong. Five of that prompt's six
steps are verification, not transport: pin an exact 40-char SHA,
confirm branch decoration, establish a green test baseline before any
change with a hard stop on mismatch, separate collection and import
errors from failures, and declare an explicit file scope. A thread that
must RUN a suite needs a working tree, and pack-repo.sh cannot supply
one -- being scratch-only and committed-read, it produces no tree by
design. If the author of the decision misread it within one thread, the
wording invites the error.
Decision: record the division explicitly. THE PACKER IS PRIMARY, since
most work is read-and-reason and that is the common case. A CHECKOUT IS
CORRECT AND NOT DEPRECATED when a thread must execute against a live
tree. Neither supersedes the other. Rule of thumb: reading is a pack,
executing is a checkout. ADR-021's scope is narrowed in wording, not in
substance -- the sparse-checkout BOOTSTRAP stays retired. transport.md
states the division; precedence.md states where a workflow prompt sits
relative to the render and instance rules.
Alternatives: leave ADR-021 as written (rejected: it produced a real
misreading by the thread that implemented it, and would mislead a
reader promoting drill's prompts into the playbook); retire
clone-and-verify (rejected: it does a job nothing else does, and the
verification discipline it encodes is worth generalizing).

ADR-033 handoff filenames: the directory carries the classification
Context: naming.md specified handoff-<FROM>-to-<TO>.md in
llm/handoffs/, which puts the word "handoff" in both the directory
and the filename. The prefix restates what the path already says. But
dropping it naively costs something real: a handoff read away from
its directory -- pasted into a chat, packed into an archive, quoted
in another document -- would no longer declare what it is. Separately,
frontmatter had been accreting fields without ever being enumerated:
N2 put dates there, and this change adds a type, with no rule saying
what fields exist. That is the accrete-without-a-rule pattern N4
warns about, applied to the playbook's own grammar.
Decision: handoff filenames take the form <FROM>-to-<TO>.md in a
handoff/ directory. The DIRECTORY carries the classification; the
file states its own type in frontmatter so it stays self-describing
when detached. naming.md additionally enumerates the permitted
frontmatter fields as a fixed set (date, type, scope, from, to,
version); a field the list does not name is a finding, not an
exception. Files already bearing the superseded form remain
classifiable and are NOT renamed in bulk -- they move to the current
form only if touched for another reason (the standing classification
note).
Alternatives: keep the prefix (rejected: it restates the path, and
the redundancy is what prompted the question); drop the prefix with
no frontmatter type (rejected: loses self-description exactly in the
cases where these files travel -- paste, archive, quotation); rename
existing handoffs in bulk to match (rejected: breaks references for
zero content gain, per the classification note).
Consequence: ADR-030's reference to "naming.md's handoff-<FROM>-to-
<TO>.md" is superseded by this decision; the kickoff template in
protocol/kickoff.md emits the current form.
