RENDER
======
date: 2026-07
version: 0.1.0
scope: how preferences reach a chat. A RENDER is a document
generated by hand from playbook preference sources (layers.md
composed with any private overlay, plus style-contract.md) and the
project's instance rules, carrying a version-and-hash stamp. The
render collapses the authoring-time precedence chain
(precedence.md): it emits only winning rules, so the model inside
a thread sees one consistent authority.

TARGETS, IN PRIORITY ORDER
  R-A. CONTEXT.md in the project's instance dir, at
       <project>/llm/CONTEXT.md (ADR-020). THE primary target:
       pointed at in repo-access chats, packed whole in plain
       chats. Ceiling 250 lines, enforced by scripts/check.sh on
       any staged CONTEXT.md. It is committed instance state,
       beside the project's other llm/ instance docs (STATUS.md,
       decisions.md, refinements.md), and serves the NEXT thread.
  R-B. CLAUDE.md and AGENTS.md beside R-A in <project>/llm/.
       SECONDARY,
       GENERATED from the same sources in the same render pass at
       zero marginal maintenance; they future-proof an eventual
       agentic environment and are never authored independently.
       Content: identical to R-A (the filename, not the content,
       is what those environments key on).
  R-C. The platform settings block: a condensed extract (persona
       and constraint items only) pasted once into a chat
       platform's standing-preferences setting. Regenerated
       whenever R-A is.

STAMP
  Line 2 of every render, exactly:
    rendered from playbook vX.Y.Z <sha> content-hash <hash8>
  where vX.Y.Z is the VERSION file's content, <sha> is the
  playbook commit the render was built from, and <hash8> is the
  first 8 characters of the sha1 of the render BODY. The body is
  everything from line 3 to end of file; lines 1-2 (header and
  stamp) are excluded so the hash is computable. Command,
  runnable from the project root:

    tail -n +3 CONTEXT.md | sha1sum | cut -c1-8

  The kickoff skeleton requires the model to STATE the stamp it
  sees at thread start. This one statement catches both failure
  modes at once: a stale version or sha (the render lags the
  playbook) and a silent hand edit (the recomputed hash differs
  from the stamped one). Context is delivered once at kickoff and
  never refreshed mid-thread, so kickoff-only checking is
  sufficient by construction.

DO-NOT-EDIT
  Line 1 of every render, exactly:
    DO NOT EDIT. Generated; fixes go to refinements, then rerender.
  A render is NEVER patched in place. A mid-thread fix reaches
  the running model as a live human message (top of the runtime
  precedence chain) and is filed as a refinement entry
  (RF-PROJ-NNN, naming.md) in the project's refinements file; the
  regenerated render serves the NEXT thread. Hand edits to a
  render are the exact failure the content hash exists to expose.

RENDER PASS, BY HAND (v0.x; automation is deferred work)
  1. Read the project's PROJECT.md instance rules and the current
     layers.md and style-contract.md at the checked-out sha.
  2. Compose: instance rules win conflicts (precedence.md chain
     1); private overlay items shadow public items by id.
  3. Write the body: instance rules and invariants first, then
     the composed preference items, then the style contract
     clauses that bind this project, each cited by its id
     (CONSTRAINT-004, S23) so refinements can target items
     precisely.
  4. Prepend lines 1-2; compute the hash with the command above;
     fill the stamp.
  5. Copy the result to CLAUDE.md and AGENTS.md unchanged, beside
     CONTEXT.md in <project>/llm/; update the platform settings
     block if persona or constraint items changed.
  6. Run scripts/check.sh (the 250-line ceiling is checked on
     staged CONTEXT.md files).
